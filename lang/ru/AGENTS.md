# Инструкции для агентов

## Коммуникация

- Владелец репозитория описывает требования на русском языке.
- Отвечайте на русском языке, если пользователь не попросит другой язык.
- Придерживайся лаконичных ответов.

## Документация

- **Всегда используй документацию и контекст проекта для ИИ агентов из папки `ai-docs/`**.
- Считайте документацию в папке `lang/ru` источником истины для описания всех документов репозитория.
- При этом в корне проекта все описания должны быть на английском языке.
- Если изменяются файлы в папке `lang/ru`, необходимо перевести эти изменения на английский язык и обновить соответствующие файлы в корне проекта.
- Локальные навыки проекта описаны в `lang/ru/SKILLS.md`, английская версия хранится в `SKILLS.md`. Если запрос совпадает с локальным навыком или его коротким названием вроде `/update-docs`, используйте его процедуру только в рамках этого репозитория.
- Дополнительные инструкции для ИИ будут размещаться в папке `ai-docs/`. Прежде чем приступать к задаче по проекту проверь все инструкции в этой папке.
- Дополнительные документы по проекту будут размещаться в папке `docs/`

## Структура репозитория

- В папке `scripts/` будут размещаться скрипты для локальной установки и применения настроек описанных библиотек и скилов.
- В папке `examples/` будут размещены примеры конфигурационных файлов.
- Инструкции для ИИ уровня репозитория должны храниться в AGENTS.md. `CLAUDE.md` должен оставаться символической ссылкой на `AGENTS.md`.

## Правила разработки

- Предпочитайте небольшие, сфокусированные изменения, соответствующие текущей структуре репозитория.
- Не перезаписывайте пользовательские изменения без явного указания.
- Перед добавлением автоматизации документируйте ожидаемую команду и необходимые переменные окружения.
- **Изменения внесённые разработчиком, запрещено заменять, удалять или откатывать без явного разрешения от человека** Всегда выводи запрос на разрешение подобных действий с описанием, что именно ты собираешься сделать.
- Если задача требует изменение кода, который уже был изменён разработчиком, агент должен сохранить эти изменения и вносить только строго необходимое изменение.
- **При изменении кода разработчиком и отказе тестов по этому коду исправляй тесты, а не новый код**. Из-за упавшего теста запрещено отменять код написанный разработчиком.

### Делай всё проще (принцип KISS)

- Руководствуйся правилом "чем проще — тем лучше" и "лучший код, интерфейс или кнопка те, которых нет". Код должен решать указанную задачу и ничего сверх.
- Реализуй то, что явно запрошено: не добавляй незатребованные возможности, настройки или «запас гибкости».
- Не обрабатывай ошибки, которые маловероятны на практике.
- Если есть возможность переписать код так, что он станет меньше, перепиши его!
- Постоянно оценивай: является ли с точки зрения опытного разработчика твоё решение переусложнённым? Сложен ли код для понимания при чтении? Если да — упрощай его.

### Соблюдай принцип DRY корректно

- Если код дублируется в нескольких местах его нужно объединить. Нужно стараться избегать дублей в коде.
- При этом, если ожидается, что будет несколько ситуаций которые потом пойдут в разном направлении, дубль разрешается. Например, создан общий код для перевода на другие языки. При этом если некоторые языки требуют особого подхода, допустимо создать дубль, чтобы избежать переусложения кода лишними проверками и ветками, который остальным языкам не нужны.

### Точечные правки

- Пиши лишь то, что нужно по запросу.
- Не исправляй попутно соседний код, комментарии, форматирование, если задача этого не требовала. Подобные улучшения допустимы толко в задачах на рефакторинг.
- Не трогай работающие механизмы.
- Всегда придерживайся указанного в проекте стиля. Даже если найдено решение в другом стиле, его нужно переписать.
- Неиспользуемый код, не связанный с задачей, удаляй только с разрешения разработчика.
- Убирай только тот код, который стал ненужным в результате твоих изменений.

### Осмысленный подход к написанию кода

- Не действуй на основании догадок. Не скрывай, если чего-то не понимаешь полностью.
- Перед тем как писать код выполни:

    1. Проведи анализ задачи от пользователя.
    2. Проговори на каких предположениях будет строиться решение.
    3. Если сомневаешься — спроси. Если возможны несколько толкований, предложи их все, не делай самостоятельный выбор.
    4. Видя более простой путь, скажи об этом.
    5. Если решение предложенное человеком не самое лучше — возрази с обоснованием.
    6. Столкнувшись с непонятным моментом, остановись и уточни всё, что вызывает двусмысленность.

### Ориентация на измеримый результат

- Определи чёткие критерии выполнения задачи. Цикл разработки длится до тех пор, пока цели не будут достигнуты.
- Преобразуй размытые задачи в проверяемые цели.
- Для многоэтапных задач сформируй план:

    Действие 1 → Проверка → Тестирование

    Действие 2 → Проверка → Тестирование

## CodeGraph

1. При работе с кодом: поиске файлов, кода, любом анализе проекта, диагностике ошибок задействуй MCP CodeGraph.

2. Если данные выглядят неактуальными, обратись к `codegraph_status`. Не запускай `codegraph sync` если активен watcher.

3. Ознакомление с кодом начинай с `codegraph_explore`.
Команду `codegraph_search` - используй только для поиска имени символа.
Команду `codegraph_node` - для извлечения полного тела символа.
Команды `codegraph_callers` и `codegraph_callees` - для анализа связей в коде.
Команду `codegraph_impact` - перед проведением рефакторинга.

4. Не открывай повторно файлы, чей исходный код CodeGraph уже нашёл, кроме ситуаций, когда требуется редактирование, проверка незакоммиченных изменений или получено предупреждение об устаревшем индексе. Применяй команы `rg`, `sed` и прямое чтение файлов лишь для неподдерживаемых индексом форматов и случаев, точного текстового поиска или в остальных случаях, когда возможностей CodeGraph оказывается недостаточно.
